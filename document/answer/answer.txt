ข้อ 1. เข้าใจปัญหาการเพิ่มขึ้นของ Itemset: Search Space
================================================================================

หากจำนวน items เป็น 30, 100 และ 200 items ตามลำดับ

-----------------------------------------------------------------------------
1.1. จำนวน itemset ที่เป็นไปได้ มีจำนวนเท่าไร ตามลำดับ
-----------------------------------------------------------------------------

จำนวน itemset ที่เป็นไปได้ทั้งหมดคำนวณจากสูตร 2^n - 1 (ไม่รวม empty set)

- n = 30 items:   2^30 - 1 = 1,073,741,823 itemsets (ประมาณ 1.07 พันล้าน)
- n = 100 items:  2^100 - 1 ≈ 1.27 × 10^30 itemsets 
- n = 200 items:  2^200 - 1 ≈ 1.61 × 10^60 itemsets

หมายเหตุ: แต่ละ item สามารถเลือกได้ 2 แบบ (อยู่หรือไม่อยู่ใน itemset) 
ดังนั้นจำนวน subset ทั้งหมดคือ 2^n แต่ลบ 1 เพราะไม่นับ empty set

-----------------------------------------------------------------------------
1.2. Search space เปลี่ยนอย่างไร ให้นักศึกษาประมาณแนวโน้ม
-----------------------------------------------------------------------------

Search space เพิ่มขึ้นแบบ Exponential (เลขชี้กำลัง)

เปรียบเทียบการเพิ่มขึ้น:
- จาก 30 → 100 items: เพิ่มขึ้นประมาณ 1.18 × 10^21 เท่า (หรือ 2^70 เท่า)
- จาก 100 → 200 items: เพิ่มขึ้นประมาณ 1.27 × 10^30 เท่า (หรือ 2^100 เท่า)
- จาก 30 → 200 items: เพิ่มขึ้นประมาณ 1.5 × 10^51 เท่า (หรือ 2^170 เท่า)

แนวโน้ม:
- Search space เพิ่มขึ้นเป็น Exponential function ของจำนวน items
- เมื่อ items เพิ่มขึ้นเชิงเส้น (linear) แต่ search space เพิ่มขึ้นแบบทวีคูณ (exponential)
- นี่คือ "Curse of Dimensionality" หรือ "Combinatorial Explosion"
- ทุกๆ การเพิ่ม 1 item ทำให้ search space เพิ่มขึ้น 2 เท่า

-----------------------------------------------------------------------------
1.3. เมื่อจำนวน item มากขึ้น มีโอกาสใช้หน่วยความจำสูงจนไม่สามารถคำนวณทั้งหมดได้
     หรือไม่ (memory explosion)
-----------------------------------------------------------------------------

ใช่ มีโอกาสสูงมากที่จะเกิด Memory Explosion

เหตุผล:
1. Memory ที่ต้องใช้เพิ่มขึ้นแบบ Exponential ตามจำนวน itemsets ที่ต้องเก็บ

2. ตัวอย่างประมาณการใช้ Memory (สมมติ itemset เฉลี่ยใช้ 100 bytes):
   - 30 items: ~100 GB (ยังพอเป็นไปได้แต่ใหญ่มาก)
   - 100 items: ~1.27 × 10^32 bytes = ไม่สามารถเก็บได้
   - 200 items: ไม่มีทางเป็นไปได้อย่างแน่นอน

3. ข้อจำกัดในทางปฏิบัติ:
   - คอมพิวเตอร์ทั่วไปมี RAM เพียง 8-64 GB
   - Server ขนาดใหญ่มี RAM ไม่เกิน 1-2 TB
   - Data Center ทั้งหมดบนโลกก็ไม่สามารถเก็บ 2^100 itemsets ได้

4. ทางแก้ไข:
   - ใช้ Pruning techniques เช่น Apriori principle (ตัด infrequent itemsets ออก)
   - ใช้ Compact Representation (เก็บเฉพาะ closed/maximal itemsets)
   - กำหนด maximum itemset size (max_k)
   - ใช้ Sampling techniques

2.1. Algorithm Comparison

- FP-Growth  → Time = 6.5506s | Memory = 0.92MB | Count = 90

-----------------------------------------------------------------------------
2.1.1. ขั้นตอนวิธีใดใช้เวลาคำนวณเร็วที่สุด เพราะอะไร
-----------------------------------------------------------------------------

คำตอบ: Apriori ใช้เวลาเร็วที่สุด (0.0247 วินาที)

เหตุผล:
1. Apriori ใน mlxtend ถูก optimize มาเป็นอย่างดีสำหรับ sparse data
2. ใช้ hash tree และ candidate pruning อย่างมีประสิทธิภาพ
3. สำหรับ dataset ขนาดเล็ก-กลาง (1,000 transactions, 30 items) 
   การสแกนข้อมูลหลายรอบของ Apriori ไม่เป็นปัญหามาก
4. FP-Growth ใช้เวลามากเพราะต้องสร้าง FP-Tree ซึ่งมี overhead สูง
   สำหรับ dataset ขนาดเล็ก

-----------------------------------------------------------------------------
2.1.2. ขั้นตอนวิธีใดใช้หน่วยความจำน้อยที่สุด
-----------------------------------------------------------------------------

คำตอบ: Compact Representation ใช้หน่วยความจำน้อยที่สุด (0.11 MB)

เหตุผล:
1. Compact เก็บเฉพาะ Top-N itemsets ที่มี support สูงสุดในแต่ละระดับ
2. ใช้ Bitset representation ซึ่งประหยัด memory
3. ทำ Pruning อย่างเข้มงวดเพื่อลดจำนวน itemsets ที่ต้องเก็บ
4. ไม่เก็บ itemsets ทั้งหมด แต่เก็บเฉพาะที่จำเป็น

เปรียบเทียบ:
- Apriori: 8.48 MB (เก็บ candidate itemsets ทั้งหมด)
- FP-Growth: 0.92 MB (เก็บ FP-Tree + conditional pattern bases)
- Compact: 0.11 MB (เก็บเฉพาะ top-N pruned itemsets)

-----------------------------------------------------------------------------
2.1.3. ขั้นตอนวิธีใดใช้หน่วยความจำใกล้คำนิยาม Information-Theoretic Minimum 
       มากที่สุด
-----------------------------------------------------------------------------

คำตอบ: Compact Representation

เหตุผล:
1. Information-Theoretic Minimum หมายถึงปริมาณข้อมูลขั้นต่ำที่จำเป็น
   ในการแทน frequent itemsets โดยไม่สูญเสียข้อมูลสำคัญ

2. Compact Representation เก็บเฉพาะ:
   - Closed Itemsets (itemsets ที่ไม่มี superset ที่มี support เท่ากัน)
   - Top-N itemsets ที่มีความสำคัญสูงสุด
   - ใช้ Bitset ซึ่งเป็น compact binary representation

3. สามารถ derive itemsets อื่นๆ ได้จาก closed itemsets
   ทำให้ไม่ต้องเก็บข้อมูลซ้ำซ้อน

4. จำนวน itemsets ที่ได้: 44 (เทียบกับ 90 ของ Apriori/FP-Growth)
   ลดลงประมาณ 51% แต่ยังคงเก็บข้อมูลที่สำคัญไว้

-----------------------------------------------------------------------------
2.1.4. ขั้นตอนวิธีใดเหมาะกับ dataset ขนาดใหญ่ มากที่สุด
-----------------------------------------------------------------------------

คำตอบ: FP-Growth (สำหรับกรณีทั่วไป) หรือ Compact Representation 
       (ถ้ายอมรับผลลัพธ์ที่ย่อได้)

เหตุผล FP-Growth เหมาะกับ dataset ขนาดใหญ่:
1. สแกนข้อมูลเพียง 2 รอบ (เทียบกับ Apriori ที่สแกนหลายรอบ)
2. ไม่ต้องสร้าง candidate itemsets
3. ใช้ FP-Tree ซึ่งเป็น compact representation ของ transactions
4. Memory usage เพิ่มขึ้นช้ากว่า Apriori เมื่อ data ขนาดใหญ่ขึ้น

เหตุผล Compact Representation เหมาะเมื่อ:
1. ต้องการผลลัพธ์ที่กระชับ
2. ยอมรับการเก็บเฉพาะ frequent itemsets ที่สำคัญที่สุด
3. มีข้อจำกัดด้าน memory อย่างมาก

จากผลการทดลอง (N=10,000):
- Apriori: Time=0.0461s, Mem=29.48MB
- FP-Growth: Time=36.7080s, Mem=7.22MB  
- Compact: Time=1.9523s, Mem=0.70MB

FP-Growth ใช้ memory น้อยกว่า Apriori 4 เท่า ที่ N=10,000
และ Memory ของ Apriori เพิ่มขึ้นเร็วกว่ามากเมื่อ N เพิ่ม

-----------------------------------------------------------------------------
2.1.5. ขั้นตอนวิธี Compact Representation มีข้อจำกัดอะไร เมื่อข้อมูลมีขนาดใหญ่ขึ้น
-----------------------------------------------------------------------------

ข้อจำกัดของ Compact Representation:

1. สูญเสียข้อมูลบางส่วน (Information Loss):
   - เก็บเฉพาะ Top-N itemsets ทำให้ itemsets ที่มี support ต่ำกว่าหายไป
   - บาง frequent itemsets ที่อาจสำคัญอาจถูกตัดออก
   - จากผล: Compact มี 44 itemsets เทียบกับ 90 ของ Apriori/FP-Growth

2. ผลลัพธ์ไม่สมบูรณ์ (Incomplete Results):
   - ไม่สามารถ enumerate ทุก frequent itemsets ได้
   - Association rules ที่สร้างจาก compact อาจไม่ครบถ้วน

3. Pruning อาจตัดสิ่งที่สำคัญออก:
   - การใช้ Apriori principle อาจตัด itemsets ที่น่าสนใจแต่มี support ต่ำ
   - Top-N strategy อาจไม่เหมาะกับทุก domain

4. ไม่เหมาะกับ dataset ที่มี long frequent itemsets:
   - ถ้า itemsets ที่ frequent มีขนาดใหญ่ การ prune อาจทำให้พลาดข้อมูลสำคัญ

5. ขึ้นอยู่กับ parameter (Top-N, max_k):
   - ต้องกำหนด parameter อย่างเหมาะสม
   - ถ้ากำหนดผิดอาจได้ผลลัพธ์ที่ไม่ดี

ข้อ 3. Association Rule
================================================================================

(ใช้ 1,000 transactions, min_support = 0.05, min_confidence = 0.3)

ผลการรัน:
- All Rules: 224 rules
- (A) Basic Rules (Support + Confidence): 121 rules
- (B) Strong/Interesting Rules (A + Lift > 1): 86 rules  
- (C) Significant Rules (B + p-value < 0.05): 25 rules

-----------------------------------------------------------------------------
3.1. ยกตัวอย่างกฎที่ถูกตัดออกในขั้นตอน A มา 1 กฎ พร้อมเหตุผล
-----------------------------------------------------------------------------

ตัวอย่างกฎที่ถูกตัดออก:
กฎ: {straw} → {bag}  
- support = ต่ำกว่า 0.05 หรือ confidence ต่ำกว่า 0.3

เหตุผลที่ถูกตัดออก:
1. Support ต่ำเกินไป: กฎนี้ปรากฏใน transactions น้อยกว่า 5% ของทั้งหมด
   หมายความว่าความสัมพันธ์นี้ไม่ได้เกิดขึ้นบ่อยพอที่จะเป็นประโยชน์
   
2. หรือ Confidence ต่ำเกินไป: แม้จะพบ straw และ bag ด้วยกัน
   แต่เมื่อมี straw โอกาสที่จะมี bag น้อยกว่า 30%
   แสดงว่าการมี straw ไม่ใช่ตัวบ่งชี้ที่ดีว่าจะมี bag

3. การกรองด้วย support และ confidence เป็นการตัดกฎที่:
   - เกิดขึ้นน้อยเกินไป (low support)
   - ไม่น่าเชื่อถือ (low confidence)

-----------------------------------------------------------------------------
3.2. ยกตัวอย่างกฎที่ถูกตัดออกในขั้นตอน B มา 1 กฎ พร้อมเหตุผล
-----------------------------------------------------------------------------

ตัวอย่างกฎที่ถูกตัดออก (จากผลการรัน):
กฎ: {soda} → {bag}
- support = 0.211
- confidence = 0.694 (69.4%)
- lift = 0.996 (≤ 1)

เหตุผลที่ถูกตัดออก:
1. Lift ≤ 1 แสดงว่า soda และ bag เป็นอิสระต่อกัน (independent)
   หรืออาจมีความสัมพันธ์เชิงลบ

2. แม้ confidence สูง (69.4%) แต่นั่นเพราะ bag เป็น item ที่ popular
   การที่มี soda ไม่ได้ทำให้โอกาสซื้อ bag เพิ่มขึ้นเลย

3. นี่คือตัวอย่างของ Spurious Correlation:
   - ดูเหมือนมีความสัมพันธ์ (confidence สูง)
   - แต่จริงๆ แล้วเป็นเพราะ bag ถูกซื้อบ่อยอยู่แล้ว (popularity effect)

4. Lift = P(Y|X) / P(Y) = 0.996 ≈ 1
   หมายความว่า P(bag|soda) ≈ P(bag) = การมี soda ไม่ส่งผลต่อการซื้อ bag

-----------------------------------------------------------------------------
3.3. ยกตัวอย่างกฎที่ถูกตัดออกในขั้นตอน C มา 1 กฎ พร้อมเหตุผล
-----------------------------------------------------------------------------

ตัวอย่างกฎที่ถูกตัดออก (จากผลการรัน):
กฎ: {beer} → {rice, bag}
- support = 0.080
- confidence = 0.323 (32.3%)
- lift = 1.182 (> 1)
- p-value = 0.0525 (≥ 0.05)

เหตุผลที่ถูกตัดออก:
1. p-value ≥ 0.05 แสดงว่าความสัมพันธ์นี้ไม่มีนัยสำคัญทางสถิติ

2. แม้ lift > 1 แต่มีโอกาสสูงกว่า 5% ที่ความสัมพันธ์นี้เกิดจากความบังเอิญ

3. การใช้ Chi-square test:
   - H0: antecedent และ consequent เป็นอิสระต่อกัน
   - p-value = 0.0525 > 0.05 → ไม่สามารถปฏิเสธ H0 ได้
   - ไม่มีหลักฐานทางสถิติเพียงพอว่ามีความสัมพันธ์จริง

4. กฎนี้อาจเป็น False Positive ที่เกิดจาก:
   - Sample size น้อย
   - Noise ใน data
   - Random variation

-----------------------------------------------------------------------------
3.4. Spurious Rules คืออะไร ส่งผลอย่างไร และการกรองมีผลดี/เสียอย่างไร
-----------------------------------------------------------------------------

Spurious Rules คืออะไร:
- กฎความสัมพันธ์ที่ดูเหมือนมีความหมาย แต่จริงๆ แล้วเกิดจากความบังเอิญ
  หรือปัจจัยแทรกซ้อนอื่น ไม่ใช่ความสัมพันธ์ที่แท้จริง

สาเหตุของ Spurious Rules:
1. Popularity Bias: items ที่ popular มักปรากฏใน rules จำนวนมาก
   ไม่ว่าจะจับคู่กับอะไร (เช่น {X} → {popular item})
2. Confounding Variables: ตัวแปรแทรกซ้อนที่ไม่ได้ควบคุม
3. Small Sample Size: ขนาดตัวอย่างน้อยทำให้เกิด random patterns
4. Multiple Testing Problem: ทดสอบหลายกฎพร้อมกันเพิ่มโอกาส false positive

ผลกระทบต่อการอธิบายความสัมพันธ์:
1. Misleading Insights: นำไปสู่ข้อสรุปที่ผิดพลาด
2. Wrong Business Decisions: ตัดสินใจทางธุรกิจผิดพลาด
   เช่น จัดวางสินค้าผิด, โปรโมชั่นที่ไม่ได้ผล
3. Wasted Resources: เสียทรัพยากรไปกับกลยุทธ์ที่ไม่ได้ผล
4. Loss of Credibility: สูญเสียความน่าเชื่อถือของการวิเคราะห์

ผลดีของการกรอง Spurious Rules:
1. ได้กฎที่น่าเชื่อถือมากขึ้น (25 กฎ จาก 121 กฎ = ~20%)
2. ลด false positives
3. ตัดสินใจได้ดีขึ้น
4. ประหยัดทรัพยากรในการ implement กฎที่ไม่มีประโยชน์

ผลเสียของการกรอง Spurious Rules:
1. อาจตัดกฎที่ดีออกไปด้วย (false negatives)
2. กฎที่เหลือน้อยลงอาจไม่เพียงพอสำหรับบาง applications
3. ต้องระวังการตั้งค่า threshold (lift > 1, p-value < 0.05)
4. บาง domain อาจต้องการกฎที่มี support ต่ำแต่สำคัญ

3.5. ทำไม 1-itemset {bag} ถึงหายไป หลังจากการกรองทั้งสามขั้นตอน
-----------------------------------------------------------------------------

เหตุผลที่ {bag} หายไป:

1. Association Rules ต้องมีทั้ง antecedent และ consequent:
   - 1-itemset ไม่สามารถสร้างกฎได้ เพราะไม่มี antecedent
   - กฎต้องมีรูปแบบ X → Y โดย X และ Y ไม่เป็น empty set

2. Rules ที่มี {bag} เป็น consequent ถูกกรองออก:
   - เนื่องจาก bag เป็น item ที่ popular มาก (support สูง)
   - ทำให้กฎ {X} → {bag} มี lift ≤ 1 (independent)
   - P(bag|X) ≈ P(bag) เพราะ bag ถูกซื้อบ่อยอยู่แล้ว

3. ตัวอย่างจากผล Spurious Rules:
   - {soda} → {bag}: lift = 0.996 (ถูกตัดใน step B)
   - {water} → {bag}: lift = 0.989 (ถูกตัดใน step B)
   - {straw} → {bag}: lift = 0.945 (ถูกตัดใน step B)

4. bag เป็นตัวอย่างของ "Popular but Uninteresting" item:
   - มี support สูง ทำให้ปรากฏในหลาย transactions
   - แต่ไม่มีความสัมพันธ์ที่แท้จริงกับ items อื่น
   - การซื้อ bag เกิดจากความต้องการทั่วไป ไม่ได้ขึ้นกับสินค้าอื่น
